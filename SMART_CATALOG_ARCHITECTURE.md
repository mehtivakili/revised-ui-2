# معماری کاتالوگ و پیشنهاد هوشمند همیار دوربین

## وضعیت این نسخه

- ۶۵ محصول Mock با الگوی دسته‌ها و مشخصات محصولات `ddcpersia.com` در PostgreSQL ذخیره شده‌اند.
- همه محصولات تصویر دارند؛ ۴۱ فایل محصول از کاتالوگ عمومی پرشیا سیستم دریافت شده و دارایی‌های AI با برچسب منشأ جدا نمایش داده می‌شوند.
- API کاتالوگ ابتدا PostgreSQL را می‌خواند و در محیط بدون دیتابیس به fixture تایپ‌شده برمی‌گردد.
- موتور پیشنهاد از دو مرحله تشکیل شده است: حذف گزینه‌های ناسازگار و سپس امتیازدهی قابل‌توضیح.
- ویزارد چهارمرحله‌ای امکان تعریف ناحیه‌ها، ذخیره پیش‌فرض محلی و ویرایش تعداد اقلام پلن نهایی را دارد.
- قیمت‌ها و موجودی‌های این نسخه نمایشی هستند و نباید برای فروش استفاده شوند.

## مدل داده

| جدول | مسئولیت |
| --- | --- |
| `catalog_brands` | برند مرجع و جلوگیری از تکرار نام برند |
| `catalog_products` | داده مشترک WooCommerce: شناسه، SKU، نام، قیمت، موجودی، URL و payload خام |
| `camera_specs` | رزولوشن، سنسور، لنز، FOV، DORI، کدک، PoE، توان، IP Rating و AI |
| `recorder_specs` | کانال، پهنای‌باند، Decode، Bay، ظرفیت هارد، RAID، کدک و PoE داخلی |
| `switch_specs` | پورت PoE، بودجه توان، uplink، Extend، مدیریت و Surge |
| `storage_specs` | ظرفیت، workload، مناسب‌بودن برای نظارت و گارانتی |
| `ups_specs` | VA، توان خروجی و زمان پشتیبانی |
| `catalog_sync_runs` | تاریخچه هر sync، تعداد ورودی/نرمال‌شده و هشدارهای کیفیت داده |

## جریان پیشنهادی برای اتصال واقعی

1. یک job زمان‌بندی‌شده در سرور، Store API را صفحه‌به‌صفحه می‌خواند؛ کلاینت موبایل مستقیماً با وردپرس صحبت نمی‌کند.
2. فیلدهای عمومی مستقیماً map و توضیحات HTML پاک‌سازی می‌شوند.
3. parserهای جداگانه برای دوربین، NVR، سوئیچ، هارد و UPS مشخصات متنی را استخراج می‌کنند.
4. واحدها به یک قرارداد ثابت تبدیل می‌شوند؛ برای نمونه `Mbps`، `W`، `TB` و `mm`.
5. رکوردهای ناقص به جای ورود خاموش به موتور پیشنهاد، در `catalog_sync_runs.warnings` ثبت و برای بازبینی علامت‌گذاری می‌شوند.
6. upsert با `woo_id` و `sku` انجام و تغییر قیمت/موجودی در نسخه بعدی در جدول تاریخچه ثبت می‌شود.
7. موتور پیشنهاد فقط جدول‌های نرمال‌شده را می‌خواند و payload خام صرفاً برای audit باقی می‌ماند.

## قرارداد API فعلی

- `GET /api/catalog?category=camera&brand=Tiandy&q=TC-C54`
- `POST /api/recommendations` با فیلدهای `projectType`، `cameraCount`، `outdoorCount`، `entrances`، `goal`، `archiveDays`، `budget` و `preferredBrand`

## منطق پیشنهاد

قیود قطعی شامل تعداد کانال، ضریب اطمینان ۲۰٪ برای پهنای‌باند و PoE، پشتیبانی رزولوشن/کدک، تعداد Bay، ظرفیت آرشیو، مقاومت فضای بیرونی و قابلیت AI موردنیاز است. پس از عبور از این قیود، امتیاز نهایی با تطابق فنی، موجودی، قیمت، گارانتی، مصرف انرژی، توسعه‌پذیری و ترجیح برند ساخته می‌شود. دلیل انتخاب و رد هر گزینه بخشی از خروجی API است.

## گام بعدی تولید

- ساخت parser و تست fixture برای هر خانواده محصول
- افزودن `product_price_history` و `inventory_history`
- اتصال خروجی محاسبه‌گر ظرفیت و شبیه‌ساز لنز به همان `ProjectBrief`
- افزودن پنل کیفیت داده برای فیلدهای ناقص، مقادیر پرت و محصولاتی که نیاز به تأیید انسانی دارند

اسکریپت seed با `npm run seed:catalog` قابل اجرای مجدد و idempotent است.

## وضعیت قابلیت‌های P2

- خروجی مهندسی اکنون Heatmap تراکم PPM، برخورد Frustum با زمین و صفحه هدف، چیدمان چند دوربینه، درصد نقاط کور و پیشنهاد اولیه محل نصب را دارد.
- هر پلن مقدار کابل مسی، Backbone فیبر، تعداد و ظرفیت Rack، Patch Panel، SFP و بار واقعی UPS را محاسبه می‌کند.
- Bitrate واقعی هر ناحیه به‌عنوان نمونه Calibration ذخیره می‌شود و پس از وجود نمونه کافی، ضریب میانه مخصوص نوع مأموریت در محاسبات بعدی اعمال می‌شود.
- پلن ذخیره‌شده در `project_calculation_versions` شامل نسخه ترتیبی، نسخه موتور، نسخه ورودی، استانداردهای مرجع، fingerprint ورودی، نقشه مهندسی و برآورد زیرساخت است؛ Local Storage نیز fallback کاربر باقی مانده است.
- خروجی PDF از مسیر چاپ مرورگر با stylesheet اختصاصی A4 تولید می‌شود.
- `POST /api/admin/catalog/sync?dryRun=true` محصولات WooCommerce را بدون تغییر دیتابیس بررسی می‌کند و `dryRun=false` برای مدیر احرازشده upsert تراکنشی انجام می‌دهد.
- ارتباط WooCommerce در کد به درخواست `GET` محدود است. تمام محصولات منتشرشده در Snapshot محلی ذخیره می‌شوند و هیچ endpoint نوشتنی WooCommerce فراخوانی نمی‌شود.
- Snapshot خام، ویژگی‌ها، تصاویر، دسته‌ها و تگ‌ها از لایه استانداردشده جدا هستند تا محصول نگاشت‌نشده نیز از دست نرود. فهرست آن‌ها از `GET /api/admin/catalog/source-products` قابل دریافت است.
- صفحه `/catalog` تمام Snapshotها را با جست‌وجو، فیلتر، pagination، Gallery و نمایش همه ویژگی‌ها ارائه می‌کند؛ نمایش دیگر به محصولات استانداردشده محدود نیست.
- تصاویر WooCommerce در `catalog_image_jobs` صف می‌شوند و Worker مستقل سرور آن‌ها را تدریجی در کش محلی دریافت می‌کند. صف پس از restart برنامه نیز از طریق `instrumentation.ts` ادامه پیدا می‌کند.
- مراحل production و قرارداد پیشنهادی ویژگی محصولات در `WOOCOMMERCE_SETUP.md` آمده است.
